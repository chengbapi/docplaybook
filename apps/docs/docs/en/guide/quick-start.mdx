# Quick Start

## 1. Install

Install DocPlaybook globally or as a dev dependency in your docs project.

**Global install:**

```bash
npm install -g docplaybook
```

**Dev dependency (recommended for team projects):**

<PackageManagerTabs variant="help" />

After installing as a dev dependency, prefix commands with your package manager's exec command (e.g. `pnpm exec docplaybook`).

## 2. Initialize your workspace

Run `init` in your docs repo root:

```bash
docplaybook init .
```

DocPlaybook will:

- detect your docs framework (Docusaurus, Rspress, VitePress, or fall back to `sibling`)
- prompt for source language and target languages
- prompt for model provider and model name
- prompt for the API key environment variable name
- run a connectivity check to confirm the model is reachable
- create `.docplaybook/config.json`
- create `.docplaybook/memories/<lang>.md` for each target language
- create `.docplaybook/playbook.md`
- create `.docplaybook/.gitignore` (excludes cache and `.env*` files)

You can also pass options directly to skip the prompts:

```bash
docplaybook init . --source en --targets zh,ja --model-kind openai --model gpt-4o --api-key-env OPENAI_API_KEY
```

If your repo already contains translated docs, `init` will suggest running `bootstrap` to seed the initial memory files from those existing translations. See [bootstrap](/guide/commands#docplaybook-bootstrap) for details.

## 3. Preview with --dry

Before making any LLM calls, run a dry-run to see what would be translated and get a token cost estimate:

```bash
docplaybook translate --dry
```

This shows every file that is stale or missing, grouped by target language, along with an estimated token count. No API calls are made.

## 4. Run your first translation

```bash
docplaybook translate
```

DocPlaybook will translate all stale and missing files. It skips any file whose source hash matches the last recorded translation, so re-running is always safe and cheap.

To translate a specific file or subdirectory:

```bash
docplaybook translate docs/guide/introduction.md
docplaybook translate docs/guide/
```

To limit to specific languages:

```bash
docplaybook translate --langs zh,ja
```

## 5. Review translations and make edits

Open the translated files and review them. Correct any terminology, phrasing, or style issues directly in the target files.

DocPlaybook will detect these edits on the next `learn` run based on file content hashes — you do not need to track diffs manually.

## 6. Run learn to capture your edits

After reviewing and editing translations, run:

```bash
docplaybook learn
```

DocPlaybook reads the current source and target files for any target that has changed since the last `learn` run. It asks the LLM to classify what it observes into four categories:

- **glossary** — deterministic term pairs to always apply
- **memory** — per-language style rules
- **playbook** — cross-language rules
- **ignore** — not a reusable rule

In interactive mode (the default when running in a terminal), each candidate is shown one at a time. You can accept it, edit it, skip it, or quit. Accepted rules are written to the appropriate knowledge asset file.

In CI or non-interactive environments, use `--no-interactive` to auto-save all candidates:

```bash
docplaybook learn --no-interactive
```

## 7. Check translation status

```bash
docplaybook status
```

This shows translation completion per language: how many files are up to date, stale, or missing. It also shows how many memory rules and glossary terms have been collected. If any files are stale or missing, it suggests running `translate`.

## 8. Add the GitHub Action for automation

To keep translations in sync automatically whenever source docs change, add the DocPlaybook GitHub Action to your repo.

Create `.github/workflows/translate.yml`:

```yaml
name: Translate docs
on:
  push:
    branches: [main]
    paths: ['docs/**/*.md']
jobs:
  translate:
    runs-on: ubuntu-latest
    permissions:
      contents: write
    steps:
      - uses: actions/checkout@v4
      - uses: docplaybook/docplaybook@v1
        with:
          working-directory: '.'
        env:
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
```

Add your API key as a repository secret under **Settings > Secrets and variables > Actions**.

See [GitHub Action](/guide/github-action) for the full setup guide including optional inputs and multi-language configuration.
