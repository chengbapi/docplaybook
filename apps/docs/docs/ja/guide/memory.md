# メモリ

DocPlaybookは翻訳ガイダンスをMarkdownで保持しており、エージェントが再利用できると同時に人間が編集可能な状態を保ちます。

## ファイル

DocPlaybookは二つのレベルのメモリを管理します:

- `.docplaybook/playbook.md`
- `.docplaybook/memories/<target>.md`

最初のものはグローバルです。二つ目は各ターゲット言語ごとに特有です。

## グローバルプレイブック

`.docplaybook/playbook.md` は言語に依存しないガイダンス用です。

以下を保持します:

- `## Voice`
- `## Protected Terms`
- `## Translation Rules`

典型的な例:

- ドキュメントは技術的かつ直接的な文体を維持するべき
- 製品名とCLIコマンドは翻訳しない
- 警告は和らげてはいけない
- Markdownの構造は変えない

## 言語メモリ

各 `.docplaybook/memories/<target>.md` ファイルは言語固有のガイダンス用です。

以下を保持します:

- `## Terminology`
- `## Style Notes`

典型的な例:

- 繰り返し出てくる技術用語の推奨訳
- 特定の英単語を無訳のままにするかどうか
- 言語固有のトーンや丁寧さの選択
- 当該言語における句読点や言い回しの好み

## メモリの使用方法

翻訳およびリンティング時に、DocPlaybookは以下を組み合わせます:

1. グローバルプレイブック
2. ターゲット言語のメモリ

その組み合わせられたコンテキストが、現在の実行における再利用可能な翻訳基準になります。

## メモリの更新方法

`docplaybook learn` はメモリが進化する主な方法です。

人間が翻訳済みドキュメントを編集したとき、学習フローは次のようになります:

1. Gitの`HEAD`にある編集されたターゲットファイルと作業ツリー内の現在のバージョンを比較します
2. ブロックの形がまだ一致している場合に、変更された翻訳可能なブロックを抽出します
3. どの編集が再利用可能な修正で、どれを無視すべきかをLLMに尋ねます
4. 言語に依存しない教訓を`playbook.md`に更新します
5. 言語固有の用語やスタイルノートを`memories/<target>.md`に更新します

これによりレビューの労力が時間とともに蓄積されていきます。

## bootstrap によるメモリの初期化

既に翻訳されたページを持つ既存のドキュメントサイトの場合、`docplaybook bootstrap --langs ...` はリポジトリ内の整列したソース/ターゲットの例から最初のメモリファイルを作成します。

この操作は、プロジェクトに既に翻訳済みのドキュメントが含まれている場合に、`init` の後に推奨される最初のステップです。

## 構成が最小限である理由

初期バージョンではメモリを意図的に小さくしています:

- グローバルルールは `playbook.md` に
- 言語固有のルールは `memories/<target>.md` に

これによりエージェントが次のことを行いやすくなります:

- ファイルを読み取る
- 安全に更新する
- 簡潔に保つ
- ルールを多数のセクションに重複して持たないようにする
