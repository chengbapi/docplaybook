# メモリ

`DocPlaybook`は翻訳ガイダンスをMarkdown形式で保持し、エージェントが再利用できると同時に人間が編集できるようにします。

## ファイル

`DocPlaybook`は2つのレベルのメモリを保持します。

- `.docplaybook/playbook.md`
- `.docplaybook/memories/<target>.md`

最初のものはグローバルで、2つ目は各ターゲット言語固有です。

`DocPlaybook`は`.docplaybook/state/*.json`も保持しますが、それらのファイルはメモリではありません。現在のブランチの処理進捗のみを記録します。

## グローバルプレイブック

`.docplaybook/playbook.md`は言語に依存しないガイダンス用です。

以下を保持します：

- `## Voice`
- `## Protected Terms`
- `## Translation Rules`

典型的な例：

- ドキュメントは技術的で直接的な表現を維持すること
- 製品名やCLIコマンドは翻訳しないこと
- 警告は和らげてはいけない
- Markdownの構造はそのまま維持すること

## 言語メモリ

各`.docplaybook/memories/<target>.md`ファイルは言語固有のガイダンス用です。

以下を保持します:

- `## Terminology`
- `## Style Notes`

典型的な例：

- 繰り返し出てくる技術用語の推奨翻訳
- 特定の英単語を翻訳せずに残すかどうか
- 言語固有のトーンや丁寧さの選択
- 当該ターゲット言語における句読点や言い回しの好み

## メモリの使用方法

翻訳やリンティングの際、`DocPlaybook`は次を組み合わせます：

1. グローバルなプレイブック
2. 対象言語のメモリ

その統合されたコンテキストは、この実行における再利用可能な翻訳基準になります。

## メモリの更新方法

`docplaybook learn`はメモリが進化する主な方法です。

人間が翻訳されたドキュメントを編集するとき、学習フローは次のとおりです：

1. 現在のターゲットファイルのハッシュが既に学習されているかを確認する
2. 現在のソースドキュメントと現在のターゲットドキュメントを読み取る
3. `LLM`にどの観察が再利用可能なガイダンスで、どれが無視すべきかを尋ねる
4. 言語に依存しない教訓を `playbook.md` に更新する
5. `memories/<target>.md` に言語固有の用語とスタイルノートを更新する

これによりレビューの工数が時間とともに蓄積されます。

## メモリとステート

この区分を明確に保つこと：

- `playbook.md`と`memories/<target>.md`はプロジェクトの知識です

ステートファイルはコミットされることがあるため、ブランチを切り替えると進捗が復元されますが、それらの内容は運用上のハッシュであり、人間が作成したガイダンスではありません。

## ブートストラップがメモリを初期化する方法

`docplaybook bootstrap --langs ...`はリポジトリ内に既に存在するソース/ターゲットの整列済みサンプルから最初のメモリファイルを作成します。

`init`の後で、プロジェクトにすでに翻訳済みのドキュメントが含まれている場合、これは推奨される最初の手順です。

## 構成が最小限である理由

最初のバージョンでは、メモリを意図的に小さく保ちます：

- グローバルルールは `playbook.md` に置きます
- 言語別ルールは `memories/<target>.md` に置きます

これによりエージェントが次のことを行いやすくなります:

- ファイルを読む
- 安全に更新する
- 簡潔に保つ
- 多くのセクションにわたってルールを重複させない
