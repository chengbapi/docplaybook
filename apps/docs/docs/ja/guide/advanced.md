# ランタイムモデル

このページでは DocPlaybook の状態駆動の実行モデルを説明します：翻訳が Git の前後のベースラインに依存せずにブロック単位の意識を保つ方法。

## 翻訳モデル

DocPlaybook はブロックレベルの構造を保持しつつ、記事単位の翻訳リクエストを使用します。

### 仕組み

1. 現在のソース Markdown をブロックに解析します。
2. 現在のソースファイルのソースハッシュを計算します。
3. そのハッシュを各ターゲット文書の最後に保存されたハッシュと比較します。
4. ソースハッシュが変わっておらずターゲットが既に存在する場合は、そのターゲットをスキップします。
5. 存在する場合は現在のターゲット文書を読み取ります。
6. ソースが変更されたかターゲットが存在しない場合、ターゲット記事に対して1回の翻訳リクエストを送信します。
7. 必要に応じて大きな記事を小さなバッチリクエストに分割します。
8. 翻訳結果を再構成し、更新されたターゲット文書を安全に書き込みます。

### このモデルを採用する理由

- ソースハッシュの状態により「このファイルを処理するべきか？」という判断が非常に軽くなります。
- ターゲット記事ごとに1回のリクエストにすることで、繰り返しのプロンプトオーバーヘッドを減らします。
- 記事単位の並列処理はブロック単位の並列より理解しやすいです。
- 大きなドキュメントは、トップレベルの「1記事1ターゲット」というメンタルモデルを変えずにチャンク分割できます。

## 学習モデル

`docplaybook learn` も状態駆動です。

各ターゲット文書について、DocPlaybook は次を行います：

1. 現在のターゲットハッシュが既に学習済みかを確認します。
2. 現在のソース文書を読み取ります。
3. 現在のターゲット文書を読み取ります。
4. LLM にどの観察事項が再利用可能なガイダンスかを尋ねます。
5. `playbook.md` に再利用可能な言語横断ルールを書き込みます。
6. `memories/<lang>.md` に再利用可能なターゲット言語ルールを書き込みます。

これにより learn は簡素になりますが、同時に「正確なレビュアーの差分を再構築する」よりも「現在のソース/ターゲットの対からルールを抽出する」に近くなります。

## ブートストラップモデル

`docplaybook bootstrap --langs ...` は既存のドキュメントサイト向けのコールドスタートパスです。

それは次を行います：

- リポジトリ内に既に存在する整列済みのソース／ターゲットペアをスキャンします。
- 最初のグローバルプレイブックを構築します。
- 選択された各ロケール向けに最初の言語メモリを構築します。

Bootstrap は複雑な隠れたベースラインを作成しません。リポジトリ内の Markdown メモリファイルのみを更新します。

## セーフティモデル

現在の設計は意図的にシンプルです：

- `.docplaybook/state/*` はソースまたはターゲットのバージョンが既に処理されたかを追跡します。
- これらの状態ファイルはブランチの進捗を示すものであり、業務知識ではありません。
- Markdown のブロック解析は構造的な単位を検出します。
- LLM が意味論的な意味と再利用可能なガイダンスを判断します。
- メモリファイルはプロジェクトの長期的な知識を保持します。

`translate` や `learn` に対する Git ベースラインの再構築ステップはありません。

実際には：

- `state` ファイルは Git に含められるので進捗は各ブランチに追随します。
- レビューでは通常ハッシュ変化の意味は無視されます。
- マージコンフリクトは通常、現在のブランチの結果を保持して必要なら `translate` や `learn` を再実行することで解決されるべきです。
