# 高度

このページでは DocPlaybook の状態駆動の実行モデルを説明します：翻訳が Git の前後のベースラインに依存せずにブロックを認識したまま動作する仕組み。

## 翻訳モデル

DocPlaybook はブロック単位の構造を維持しつつ、記事単位の翻訳リクエストを使用します。

### 動作の仕組み

1. 現在のソース Markdown をブロックにパースする。
2. 現在のソースファイルのソースハッシュを算出する。
3. そのハッシュを各ターゲット文書の最後に保存されたハッシュと比較する。
4. ソースハッシュが変わっておらずターゲットが既に存在する場合はそのターゲットをスキップする。
5. ターゲット文書が存在する場合は現在のターゲット文書を読み込む。
6. ソースが変更された場合、またはターゲットが存在しない場合はターゲット記事に対して翻訳リクエストを1回送る。
7. 大きな記事は必要に応じて小さなバッチリクエストに分割する。
8. 翻訳結果を再構成し、更新されたターゲット文書を安全に書き込む。

### なぜこのモデルか

- ソースハッシュの状態により、「このファイルを処理すべきか？」という判定が非常に安価になる。
- ターゲット記事ごとに1回のリクエストにすることで、繰り返しのプロンプトオーバーヘッドを削減する。
- 記事レベルの並行処理はブロックレベルの並行処理より理解しやすい。
- 大きなドキュメントはトップレベルの「1ターゲット記事」モデルを変えずにチャンク化できる。

## 学習モデル

`docplaybook learn` も状態駆動で動作します。

各ターゲット文書について、DocPlaybook は次を行います：

1. 現在のターゲットハッシュが既に学習済みかをチェックする
2. 現在のソース文書を読み取る
3. 現在のターゲット文書を読み取る
4. LLM にどの観察結果が再利用可能なガイダンスかを問い合せる
5. `playbook.md` に再利用可能な言語横断ルールを書き込む
6. `memories/<lang>.md` に再利用可能なターゲット言語ルールを書き込む

これにより `learn` は単純化されますが、同時に「現在のソース/ターゲットペアからルールを抽出する」方に近くなり、「レビュワーの正確な差分を再構築する」方ではないということを意味します。

## ブートストラップモデル

`docplaybook bootstrap --langs ...` は既存のドキュメントサイトに対するコールドスタート用のパスです。

それは次のことを行います：

- リポジトリ内に既に存在する整列済みのソース/ターゲットペアをスキャンする
- 初回のグローバル playbook を構築する
- 選択された各ロケールについて初回の言語メモリを構築する

Bootstrap は複雑な隠れたベースラインを作成しません。リポジトリ内の Markdown メモリファイルのみを更新します。

## 安全モデル

現在の設計は意図的にシンプルです：

- `.docplaybook/state/*` はソースまたはターゲットのバージョンが既に処理済みかを追跡する
- Markdown のブロック解析が構造的な単位を見つける
- LLM が意味的な内容と再利用可能なガイダンスを判断する
- メモリファイルが長期的なプロジェクト知識を永続化する

`translate` や `learn` に対して Git のベースライン再構築ステップはありません。
