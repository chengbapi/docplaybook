# 上級

このページではDocPlaybookの背後にあるGitファーストの実行モデルについて説明します：翻訳が隠れたランタイムのベースラインに頼らずにどのようにブロック単位で認識され続けるかを示します。

## 翻訳モデル

DocPlaybookはブロック単位の構造を保持しつつ、記事単位の翻訳リクエストを使用します。

### 仕組み

1. 現在のソースMarkdownをブロックに解析する。
2. Gitの`HEAD`から同じソースファイルを読み取る。
3. 2つのソーススナップショットを比較して、変更された翻訳可能なブロックを見つける。
4. 存在する場合、現在のターゲット文書を読み取る。
5. ソースとターゲットの構造がまだ一致している場合は、変更のないターゲットブロックを再利用する。
6. ターゲット記事に対して1回の翻訳リクエストを送る。
7. 返ってきた結果をブロックIDで分割し、更新されたターゲット箇所のみを書き込む。

### このモデルを採用する理由

- Gitは変更前/変更後の唯一の権威であるため、変更の出所を簡単に検査・レビューできる。
- ブロック単位の差分により置換が正確になる。
- ターゲット記事ごとに1回のリクエストで、繰り返しのプロンプトオーバーヘッドを減らす。
- 記事単位の並行処理はブロック単位の並行処理より考慮しやすい。

## 学習モデル

`docplaybook learn`もGitファーストです。

変更された翻訳済みファイルごとに、DocPlaybookは次を行います。

1. Gitの`HEAD`からターゲットファイルを読み取る
2. カレントのワーキングツリーのバージョンを読み取る
3. 両方をMarkdownのブロックに解析する
4. 構造がまだ一致している場合に、変更された翻訳可能なブロックを抽出する
5. どの編集が再利用可能なガイダンスかをLLMに問う
6. 再利用可能な言語横断ルールを`playbook.md`に書き込む
7. 再利用可能なターゲット言語ルールを`memories/<lang>.md`に書き込む

ブロック構造が大きく変化している場合、安定しない整合を推測しようとする代わりに、安全のためそのファイルはスキップされます。

## ブートストラップモデル

`docplaybook bootstrap --langs ...`は既存のドキュメントサイト向けのコールドスタート経路です。

それは次のことを行います:

- リポジトリに既に存在する整列済みのソース/ターゲットペアをスキャンする
- 最初のグローバルプレイブックを構築する
- 選択したロケールごとに最初の言語メモリを構築する

Bootstrapは隠れたベースライン状態を作成しません。リポジトリ内のMarkdownメモリファイルのみを更新します。

## 安全性モデル

現在の設計はあえてシンプルです:

- Gitはソースおよびターゲットファイルの履歴を追跡する
- Markdownのブロック解析が構造単位を見つける
- LLMが意味的な解釈と再利用可能なガイダンスを判断する
- メモリファイルはプロジェクトの知識を長期的に保持する

ブランチやチームメンバー間で同期を保つための追加のランタイムディレクトリは存在しません。
