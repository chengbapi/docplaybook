# クイックスタート

## プロジェクトにインストール

既存のドキュメントプロジェクトに`DocPlaybook`を統合する場合、開発依存としてインストールするのが通常最適です。

<PackageManagerTabs variant="help" />

## プロジェクトの初期化

インストール後、最初の実際の手順は次のとおりです：

```bash
pnpm exec docplaybook init .
```

`DocPlaybook`は、初期化がほとんどのプロジェクトで同様になるよう設計されています。既にドキュメントが`Docusaurus`、`Rspress`、`VitePress`を使用している場合、`DocPlaybook`はそれを検出して内部で対応するパス規約を適用します。

サポートされたドキュメントフレームワークが検出されない場合、`sibling`にフォールバックします。

`init`中に、`DocPlaybook`は次のことを行います：

- 可能な場合はワークスペースのレイアウトを検出する
- モデルプロバイダとモデルを選択する
- 必要な認証情報を収集する
- モデルの接続性をテストする
- ソース言語を検出する
- 対象言語を尋ねる
- `.docplaybook/config.json`を作成する
- `.docplaybook/memories/*.md`を作成する
- `.docplaybook/playbook.md`を作成する

既に翻訳済みのターゲットドキュメントが存在する場合、`init`は既存のドキュメントから最初の`playbook`と言語用の`memories`を推測するために、`bootstrap`の実行を提案します。

翻訳と学習の実行が始まると、`DocPlaybook`は`.docplaybook/state/*.json`も維持します。これらのファイルはブランチの進行状況を追跡し、ブランチとともにコミットすることを想定しています。

## 既存の翻訳済みドキュメントのブートストラップ

リポジトリにすでに対応する翻訳ドキュメントが含まれている場合、`init`の後で`bootstrap`を実行してください：

```bash
pnpm exec docplaybook bootstrap . --langs ja
```

`bootstrap`を使用して、既存の翻訳から初期プロジェクトのガイダンスを推測します：

- `.docplaybook/playbook.md`
- `.docplaybook/memories/<lang>.md`

その後は通常、`translate`、`learn`、`lint`を使用してメンテナンスを行います。

## プロバイダーの設定

`DocPlaybook`はAI対応でLLMベースのため、プロバイダーの設定は初期化の一部です。

初期化中に、次のことを行います：

- プロバイダーを選択する
- モデルIDを選択する
- 認証情報を提供する
- 接続チェックを実行する

チームに合ったプロバイダーを選択できます。`DocPlaybook`は特定のプロバイダーファミリーを必須としません。

チームのワークフローでは、多くのプロジェクトがプロバイダーとモデルを設定ファイルで固定し、ローカル実行とCI実行の一貫性を保ちます。

## レイアウトとフレームワークの規約

ユーザーフローは同じですが、出力レイアウトはDocPlaybookが検出した内容によって変わります：

```text
sibling:
guide.md
guide.en.md
guide.ja.md

docusaurus:
docs/guide/intro.md
i18n/en/docusaurus-plugin-content-docs/current/guide/intro.md

rspress:
docs/en/guide/intro.md
docs/ja/guide/intro.md

vitepress:
docs/guide/intro.md
docs/en/guide/intro.md
```

### Docusaurus

`DocPlaybook`が`Docusaurus`を検出すると、翻訳されたドキュメントを次に書き込みます：

```text
i18n/<locale>/docusaurus-plugin-content-docs/current/
```

これは公式のDocusaurusドキュメントのi18n構成に従っています。

### Rspress

`DocPlaybook`が`Rspress`を検出すると、翻訳されたドキュメントを次に書き込みます：

```text
docs/<locale>/
```

これはRspress向けのDocPlaybook統合の慣例として理解してください。

### VitePress

`DocPlaybook`が`VitePress`を検出すると、翻訳されたドキュメントを次に書き込みます：

```text
docs/<locale>/
```

これは`VitePress`プロジェクトで使われる一般的なローカライズされたドキュメントレイアウトに従っています。

### `sibling`

`DocPlaybook`はフォールバックとして`sibling`を使用し、翻訳ファイルはソースファイルの隣に配置されます：

```text
guide.md
guide.en.md
guide.ja.md
```
