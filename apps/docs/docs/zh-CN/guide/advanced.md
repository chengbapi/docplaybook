# 运行机制

本页介绍 `DocPlaybook` 背后的基于状态的执行机制：说明在不依赖 Git 变更基线（前/后）的情况下，翻译如何保持对文档块的感知。

## 翻译机制

`DocPlaybook` 保持块级结构，但使用文章级别的翻译请求。

### 工作原理

1. 解析当前的源 Markdown 为若干块。
2. 为当前源文件计算源哈希。
3. 将该哈希与每个目标文档上次保存的哈希进行比较。
4. 当源哈希未改变且目标已存在时跳过该目标。
5. 如果存在，则读取当前目标文档。
6. 当源已更改或目标缺失时，为该目标文章发送一次翻译请求。
7. 在需要时将大型文章拆分为更小的批量请求。
8. 重新组装翻译结果并安全地写入更新后的目标文档。

### 选择此机制的原因

- 源哈希状态使得“是否需要处理此文件？”的判断非常低成本。
- 对每个目标文章一次请求可减少重复提示的开销。
- 文章级并发比块级并发更易于理解。
- 可以在不改变顶层“一篇目标文章”心智模型的情况下对大型文档进行分块。

## 学习机制

`docplaybook learn` 也是基于状态的。

对于每个目标文档，DocPlaybook：

1. 检查当前目标哈希是否已被学习过。
2. 读取当前源文档。
3. 读取当前目标文档。
4. 询问 LLM 哪些观察结果是可复用的指导。
5. 将可复用的跨语言规则写入 `playbook.md`。
6. 将可复用的目标语言规则写入 `memories/<lang>.md`。

这使得 `learn` 更简单，但也意味着它更接近于“从当前源/目标对中提取规则”，而不是“重建审阅者的精确差异”。

## 引导机制

`docplaybook bootstrap --langs ...` 是现有文档站点的冷启动路径。

它会：

- 扫描仓库中已存在的对齐源/目标对。
- 构建第一个全局 playbook。
- 为每个选定的语言构建首个语言记忆。

引导不会创建复杂的隐藏基线。它只会更新仓库内的 Markdown 记忆文件。

## 安全机制

当前设计是刻意保持简洁的：

- `.docplaybook/state/*` 跟踪源或目标版本是否已被处理。
- 这些状态文件表示分支进度，而非业务知识。
- Markdown 块解析用于发现结构单元。
- LLM 判断语义含义和可复用的指导。
- 记忆文件用于持久化长期的项目知识。

对于 `translate` 或 `learn`，不存在 Git 基线重建步骤。

实际上：

- 允许将 `state` 文件提交到 Git，因此进度会随每个分支保存。
- 审阅通常忽略哈希变化的含义。
- 合并冲突通常应通过保留当前分支的结果来解决，如有需要可重新运行 `translate` 或 `learn`。
