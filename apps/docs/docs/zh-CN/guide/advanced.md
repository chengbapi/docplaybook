# 高级

本页介绍 DocPlaybook 背后的以 Git 为先的执行模型：说明翻译如何在不依赖隐藏运行时基线的情况下保持对区块的感知。

## 翻译模型

DocPlaybook 保持区块级结构，但使用文章级的翻译请求。

### 工作原理

1. 将当前源 Markdown 解析为区块。
2. 从 Git 的 `HEAD` 读取相同的源文件。
3. 比较两个源快照并找出已更改的可翻译区块。
4. 读取当前的目标文档（如果存在）。
5. 当源和目标的结构仍然一致时，重用未改变的目标区块。
6. 对目标文章发送一次翻译请求。
7. 按区块 id 拆分返回结果，并仅写入已更新的目标位置。

### 采用此模型的原因

- Git 是唯一的前后状态权威，因此变更来源易于检查和审查。
- 区块级 diff 保持替换精确。
- 每篇目标文章一次请求可减少重复提示的开销。
- 与区块级并发相比，文章级并发更容易推理和管理。

## 学习模型

`docplaybook learn` 同样以 Git 为先。

对于每个已更改的已翻译文件，DocPlaybook 会：

1. 从 Git 的 `HEAD` 读取目标文件
2. 读取当前工作树版本
3. 将两者解析为 Markdown 区块
4. 在结构仍然对齐时提取已更改的可翻译区块
5. 询问 LLM 哪些编辑可以作为可重用的指导
6. 将可重用的跨语言规则写入 `playbook.md`
7. 将可重用的目标语言规则写入 `memories/<lang>.md`

如果区块结构变化过大，为了安全起见该文件将被跳过，而不是尝试推断不稳定的对齐关系。

## 引导模型

`docplaybook bootstrap --langs ...` 是现有文档站点的冷启动路径。

它会：

- 扫描仓库中已存在的已对齐的源/目标对
- 构建首个全局 playbook
- 为每个选定的语言构建首个语言记忆

引导过程不会创建隐藏的基线状态。它只会更新仓库内的 Markdown 记忆文件。

## 安全模型

当前设计有意保持简单：

- Git 跟踪源文件和目标文件的历史
- Markdown 区块解析发现结构单元
- LLM 判断语义含义和可重用的指导
- 记忆文件持久保存长期的项目知识

没有额外的运行时目录需要在分支或团队成员之间保持同步。
