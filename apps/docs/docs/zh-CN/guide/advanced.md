# 高级

本页介绍 DocPlaybook 背后的状态驱动执行模型：以及翻译如何在不依赖 Git 前后基线的情况下保持块级感知。

## 翻译模型

DocPlaybook 保持块级结构，但使用文章级的翻译请求。

### 工作原理

1. 将当前源 Markdown 解析为块。
2. 计算当前源文件的源哈希。
3. 将该哈希与每个目标文档最后保存的哈希进行比较。
4. 如果源哈希未改变且目标已存在，则跳过该目标。
5. 如果存在，读取当前目标文档。
6. 当源已更改或目标缺失时，向目标文章发送一次翻译请求。
7. 在需要时将大型文章拆分成更小的批量请求。
8. 重新组装翻译结果并安全地写入更新后的目标文档。

### 为什么采用此模型

- 源哈希状态使“我是否应处理此文件？”的判断开销很小。
- 每个目标文章一次请求可减少重复提示的开销。
- 与块级并发相比，文章级并发更易推理。
- 可以在不改变顶层“每个目标文章一个请求”心智模型的情况下对大型文档进行分块。

## 学习模型

`docplaybook learn` 也采用状态驱动。

对于每个目标文档，DocPlaybook 会：

1. 检查当前目标哈希是否已被学习过
2. 读取当前源文档
3. 读取当前目标文档
4. 询问 LLM 哪些观察结果可作为可重用的指导
5. 将可重用的跨语言规则写入 `playbook.md`
6. 将可重用的目标语言规则写入 `memories/<lang>.md`

这使得 learn 更简单，但也意味着它更接近于“从当前源/目标对中提取规则”，而非“重建审阅者的精确差异”。

## 引导模型

`docplaybook bootstrap --langs ...` 是现有文档站点的冷启动路径。

它会：

- 扫描仓库中已存在的已对齐的源/目标对
- 构建第一个全局 playbook
- 为每个选择的语言构建第一个语言记忆

Bootstrap 不会创建复杂的隐藏基线。它仅会更新仓库内的 Markdown 记忆文件。

## 安全模型

当前设计故意保持简单：

- `.docplaybook/state/*` 跟踪源或目标版本是否已被处理
- Markdown 块解析找到结构单元
- LLM 判断语义含义和可重用的指导
- 记忆文件持久化长期的项目知识

对于 `translate` 或 `learn`，没有 Git 基线重建步骤。
