# 代理

DocPlaybook 围绕 LLM 代理构建，而不仅仅是一次性的翻译调用。

这些代理负责翻译文档、从审校中学习、维护可复用的指导文件，并根据这些指导对翻译输出进行 lint 检查。

## 代理维护的内容

DocPlaybook 保持两层由 AI 维护的指导：

- `.docplaybook/playbook.md`
- `.docplaybook/memories/<target>.md`

代理会随时间更新这些文件，以便后续的翻译和 lint 运行可以重用先前的决策。

## 代理职责

### 翻译

翻译代理：

- 读取源文档
- 读取全局 playbook 和目标语言记忆
- 仅翻译需要重新生成的目标位置
- 将其余已翻译的文档原样保留

### 学习

学习代理：

- 将 Git `HEAD` 中的已翻译文件与当前工作区版本进行比较
- 在结构仍然对齐时，提取已更改的可翻译块
- 让 LLM 判定这些编辑是否为可复用的更正
- 使用与语言无关的规则更新 `.docplaybook/playbook.md`
- 使用语言特定的术语和风格说明更新 `.docplaybook/memories/<target>.md`

### 校验

校验代理：

- 读取源文档
- 读取翻译后的文档
- 读取全局 playbook 和目标语言记忆
- 在多个维度上对质量进行评分
- 输出明确的发现
- 可使用 `--fix` 应用安全修复

## 全局与语言特定的指导

代理故意保持这种划分：

- `playbook.md`
  - 与语言无关的指导
  - 共享语气
  - 受保护术语
  - 项目范围的翻译规则
- `memories/<target>.md`
  - 语言特定的术语
  - 语言特定的风格说明

这能让初始版本保持简洁，同时让代理随着时间学习有用的规则。

## 为什么采用此模型

这种基于代理的结构有助于 DocPlaybook 在反复运行中保持实用性：

- 翻译变得更一致
- 人工审校的投入得以累积而不会丢失
- lint 有更明确的评判标准
- 指导可编辑，因为它以 Markdown 存储
