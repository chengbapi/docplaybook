# 代理

DocPlaybook 围绕 LLM 代理构建，而不仅是一次性的翻译调用。

这些代理负责翻译文档、从审阅中学习、维护可复用的指导文件，并按照这些指导对翻译输出进行 lint 检查。

## 代理维护的内容

DocPlaybook 保持两层由 AI 维护的指导：

- `.docplaybook/playbook.md`
- `.docplaybook/memories/<target>.md`

代理会随着时间更新这些文件，以便未来的翻译和 lint 运行可以重用先前的决策。

## 代理职责

### 翻译

翻译代理：

- 读取源文档
- 读取全局 playbook 和目标语言记忆
- 检查每个目标文档的源哈希是否已更改
- 跳过已经是最新的目标文档
- 当源已更改或目标缺失时，刷新整个目标文档

### 学习

学习代理：

- 检查自上次学习运行以来目标哈希是否发生变化
- 跳过已在当前版本学习过的目标
- 让 LLM 从当前的源/目标对中提取可复用的指导
- 使用语言无关的规则更新 `.docplaybook/playbook.md`
- 使用语言特定的术语和样式说明更新 `.docplaybook/memories/<target>.md`

### Lint（静态检查）

Lint 代理：

- 读取源文档
- 读取已翻译文档
- 读取全局 playbook 和目标语言记忆
- 在多个维度上评分质量
- 报告明确的发现
- 可以使用 `--fix` 应用安全修复

## 全局指导与语言特定指导

代理有意保持这种划分：

- `playbook.md`
  - 语言无关的指导
  - 统一的语调
  - 受保护的术语
  - 项目范围的翻译规则
- `memories/<target>.md`
  - 语言特定的术语
  - 语言特定的样式说明

这使得初始版本保持简单，同时允许代理随着时间学习有用规则。

## 为什么采用这种模型

这种基于代理的结构有助于 DocPlaybook 在多次运行中保持有用：

- 翻译变得更一致
- 人工审阅的投入会累积而不是流失
- lint 有更明确的评判标准
- 指导保持可编辑，因为它以 Markdown 存储
